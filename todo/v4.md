### V2-1 В· AST-Aware Chunking

**File:** `crates/mccp-indexer/src/chunker.rs`
**Depends on:** nothing
**Blocks:** V2-2 (snapshot quality), V4-3 (structure analysis uses same tree-sitter parse)

**Problem:** `chunk_with_parser()` falls back to `chunk_simple()` (line-count split).
Methods and functions are cut in half, destroying semantic coherence.

#### Splittable node types per language

| Language       | Node types |
|----------------|------------|
| Rust           | `function_item`, `impl_item`, `struct_item`, `enum_item`, `trait_item`, `mod_item` |
| TypeScript/JS  | `function_declaration`, `arrow_function`, `class_declaration`, `method_definition`, `export_statement`, `interface_declaration`, `type_alias_declaration` |
| Python         | `function_definition`, `class_definition`, `decorated_definition`, `async_function_definition` |
| Go             | `function_declaration`, `method_declaration`, `type_declaration`, `var_declaration`, `const_declaration` |
| Java           | `method_declaration`, `class_declaration`, `interface_declaration`, `constructor_declaration` |
| C/C++          | `function_definition`, `class_specifier`, `namespace_definition` |
| C#             | `method_declaration`, `class_declaration`, `interface_declaration`, `struct_declaration`, `enum_declaration` |

#### Key functions to implement

```rust
// chunker.rs
fn lang_config(lang: Language) -> Option<LangConfig>
fn chunk_with_parser(&self, file: &SourceFile) -> Vec<Chunk>
fn collect_splittable_nodes(node, src, splittable, out, path, lang)
fn refine_large_chunks(&self, chunks: Vec<RawChunk>) -> Vec<RawChunk>
fn split_large_chunk_by_line(&self, chunk: RawChunk, max_chars: usize) -> Vec<RawChunk>
fn apply_overlap_ts_style(&self, chunks: Vec<RawChunk>) -> Vec<Chunk>
```

**IMPORTANT:** `collect_splittable_nodes` must NOT recurse into a matched node's children вЂ”
they are part of that chunk, not separate chunks.

#### Cargo.toml (workspace вЂ” confirm present)
```toml
tree-sitter            = "0.21"
tree-sitter-rust       = "0.21"
tree-sitter-typescript = "0.21"
tree-sitter-python     = "0.21"
tree-sitter-go         = "0.21"
tree-sitter-java       = "0.21"
```

- [ ] Replace `chunk_with_parser` stub with real tree-sitter parse
- [ ] Implement `collect_splittable_nodes` (DFS, no child recurse into matched nodes)
- [ ] Implement `refine_large_chunks` + `split_large_chunk_by_line`
- [ ] Implement `apply_overlap_ts_style`

- [ ] Test: oversized function (>512 tokens) в†’ subdivided into sub-chunks

### V2-2 В· MerkleDAG Incremental Sync

**File:** `crates/mccp-indexer/src/merkle.rs` (new), `pipeline.rs`, `crates/mccp-storage/src/persistence.rs`
**Depends on:** V2-4 (ignore rules must be same as the snapshot walker)
**Blocks:** nothing

**Problem:** Current O(n) polling re-hashes every file on every cycle even when nothing changed.

#### Key types and functions

```rust
// merkle.rs
pub struct MerkleSnapshot { root_hash: String, file_hashes: BTreeMap<String, String> }
pub struct SnapshotDiff { added: Vec<String>, removed: Vec<String>, modified: Vec<String> }

impl MerkleSnapshot {
    pub fn build(file_hashes: BTreeMap<String, String>) -> Self
    pub fn diff(&self, new: &MerkleSnapshot) -> Option<SnapshotDiff>
}

impl SnapshotDiff {
    pub fn is_empty(&self) -> bool
    pub fn total_changed(&self) -> usize
}

// persistence.rs
pub async fn save_snapshot(project_id: &str, snap: &MerkleSnapshot) -> Result<()>
pub async fn load_snapshot(project_id: &str) -> Result<Option<MerkleSnapshot>>
fn snapshot_path(project_id: &str) -> PathBuf   // ~/.mccp/data/mccp/{id}/snapshot.json
```

#### Pipeline integration points

Replace `file_hash_cache: dashmap::DashMap<String, String>` with:
```rust
snapshot: Arc<tokio::sync::RwLock<Option<MerkleSnapshot>>>
```

On every index cycle: build new snapshot в†’ diff в†’ process only changed files.

### V4-1 В· Codebase Intelligence System

**Files:** `crates/mccp-indexer/src/code_intel.rs` (new), `crates/mccp-core/src/schema.rs` (new)
**Depends on:** V2-1 (tree-sitter already wired), V2-7

**Purpose:** Give mccp (and its AI agents) a queryable map of the entire codebase:
which functions exist, what they call, which variables they use, where they are defined,
and where they are referenced. This data prevents duplicate implementations and circular bugs.

#### Why not just use an existing tool?

We support multiple external tools (see В§Pluggable Analyzers below), but we also maintain
a built-in lightweight analyzer so the system works offline without extra installations.

#### Pluggable Analyzer System

Agents and users can choose their code analysis backend. mccp wraps each in a uniform adapter.

| **cargo-modules**  | Rust modules only  | MIT       | `cargo install cargo-modules`; visualizes crate/module graph |
| **tree-sitter**    | All languages      | MIT       | Built-in, already a dep; used for our native analyzer |

#### Schema вЂ” `CodeIntelSnapshot`

```rust
// crates/mccp-core/src/schema.rs

/// Top-level snapshot of codebase structure, serialized to
/// ~/.mccp/data/mccp/{project_id}/code_intel.json
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct CodeIntelSnapshot {
    pub project_id:  String,
    pub built_at:    i64,                        // unix timestamp
    pub symbols:     Vec<SymbolDef>,
    pub call_edges:  Vec<CallEdge>,              // function в†’ function calls
    pub use_edges:   Vec<UseEdge>,               // function в†’ variable reads/writes
    pub import_edges: Vec<ImportEdge>,           // file в†’ file imports
}

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct SymbolDef {
    pub id:           String,  // "{file}::{name}::{kind}" вЂ” stable across renames
    pub name:         String,
    pub kind:         SymbolKind,
    pub file:         String,
    pub start_line:   u32,
    pub end_line:     u32,
    pub visibility:   Visibility,
    pub doc_comment:  Option<String>,
    pub references:   Vec<SymbolRef>,  // all call/use sites
}

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub enum SymbolKind { Function, Method, Struct, Enum, Trait, Variable, Constant, Module, Type }

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub enum Visibility { Public, Private, Crate, Super }

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct SymbolRef { pub file: String, pub line: u32, pub context: String }

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct CallEdge { pub caller: String, pub callee: String }   // symbol ids

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct UseEdge { pub user: String, pub used: String, pub kind: UseKind }

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub enum UseKind { Read, Write, Move, Borrow }

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct ImportEdge { pub from_file: String, pub to_file: String, pub symbol: Option<String> }
```

### V4-2 В· Circular Dependency Guard

**Files:** `crates/mccp-indexer/src/cycle_detector.rs` (new), `crates/mccp-core/src/schema.rs`
**Depends on:** V4-1 (needs `CodeIntelSnapshot` with call/import edges)

**Purpose:** Detect and report circular function call chains and circular module imports
before they cause stack overflows or compilation failures.

#### Algorithm

Use **Depth-First Search with a color-marking algorithm** (standard cycle detection in directed graphs):
- WHITE (0) = unvisited
- GRAY (1) = in current DFS path (back-edge = cycle!)
- BLACK (2) = fully processed

This is O(V + E) вЂ” safe for large codebases.

#### Key types and functions

```rust
// cycle_detector.rs

#[derive(Debug, Clone, serde::Serialize)]
pub struct CycleReport {
    pub call_cycles:   Vec<Vec<String>>,   // each Vec is a cycle: [a, b, c, a]
    pub import_cycles: Vec<Vec<String>>,   // file-level circular imports
}

pub struct CycleDetector;
impl CycleDetector {
    pub fn detect_call_cycles(snapshot: &CodeIntelSnapshot) -> CycleReport
    pub fn detect_import_cycles(snapshot: &CodeIntelSnapshot) -> CycleReport

    fn dfs_call(
        node: &str,
        edges: &HashMap<String, Vec<String>>,
        colors: &mut HashMap<String, u8>,
        path:   &mut Vec<String>,
        cycles: &mut Vec<Vec<String>>,
    )
}
```

#### Integration points

1. Run cycle detection after every full index build.
2. Emit warnings in the indexer log for any detected cycles.
3. Expose via `GET /v1/code_intel/cycles` REST endpoint.
4. Surface in `mccp status` output: `Cycles: 0 call / 0 import`.
5. The MCP `query` tool can include cycle warnings in context when relevant.

#### Stack overflow prevention

For functions in a call cycle, add a `#[cycle_warning]` annotation in the `SymbolDef`
and include a note in the chunk content: `// WARNING: this function is in a call cycle`.
This prevents agents from deepening cycles unknowingly.
