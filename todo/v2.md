# mccp v2 — Migration & Improvement Guide

> **Source of truth for this analysis:**
> - `AGENT.md` — the architectural specification for mccp
> - `todo/claude-context-master/` — reference implementation (TypeScript, Zilliz/Milvus) from which algorithms are adapted
> - `crates/` — current Rust implementation (~8 700 lines across 6 crates)

---

## Executive Summary

mccp has a solid foundation: trait-based provider abstraction, symbol extractor, configurable chunker, BFS graph engine, and a working MCP server scaffold. However several critical pipeline components are **stubs** (`// TODO` or regex-only implementations), and claude-context-master contains proven algorithms that map directly onto the mccp architecture.

This document covers **seven upgrade areas** ordered by impact:

| # | Area | Current state | Algorithm source | Impact |
|---|------|--------------|-----------------|--------|
| 1 | AST-aware chunking | stub (`chunk_with_parser` falls back to line split) | `ast-splitter.ts` | High — better retrieval quality |
| 2 | MerkleDAG incremental sync | simple SHA-256 per file | `merkle.ts` + `synchronizer.ts` | High — O(1) change detection |
| 3 | Hybrid search (dense + sparse/BM25) | dense-only Qdrant search | `milvus-vectordb.ts` hybrid path | High — recall for exact symbols |
| 4 | `.gitignore`-aware file filtering | hardcoded defaults only | `context.ts` `loadIgnorePatterns` | Medium — correctness |
| 5 | Embedding dimension auto-detection | manual config required | `ollama-embedding.ts` | Medium — DX |
| 6 | Progress callback / streaming index status | no per-file progress | `context.ts` `progressCallback` | Medium — observability |
| 7 | mccp-server crate (missing from workspace) | Cargo.toml references it but no `[[bin]]` | AGENT.md daemon spec | Critical — needed to ship |

## Area 1 — AST-Aware Chunking

### Current gap

`crates/mccp-indexer/src/chunker.rs` → `chunk_with_parser()`:
```rust
fn chunk_with_parser(&self, file: &SourceFile) -> Vec<Chunk> {
    // For now, fall back to simple chunking
    // TODO: Implement tree-sitter integration
    self.chunk_simple(file)
}
```

All files, including Rust/TypeScript/Python, go through the line-count-based `chunk_simple()`. This creates chunks that split methods in the middle, breaking semantic coherence and degrading retrieval quality.

### Reference algorithm (`ast-splitter.ts`)

The claude-context AST splitter:

1. Maps each language to its tree-sitter grammar and a set of **splittable node types** (function declarations, class declarations, method definitions, etc.)
2. Walks the AST with DFS, collecting subtrees whose root matches a splittable type
3. If a collected node's text exceeds `chunkSize`, subdivides line-by-line into sub-chunks
4. Applies **character-level overlap** from the *previous* chunk's tail — not a full re-parse, just string prepend

```
SPLITTABLE_NODE_TYPES (per language):
  rust        → function_item, impl_item, struct_item, enum_item, trait_item, mod_item
  typescript  → function_declaration, arrow_function, class_declaration,
                method_definition, export_statement, interface_declaration,
                type_alias_declaration
  python      → function_definition, class_definition, decorated_definition,
                async_function_definition
  java        → method_declaration, class_declaration, interface_declaration,
                constructor_declaration
  go          → function_declaration, method_declaration, type_declaration,
                var_declaration, const_declaration
  c/cpp       → function_definition, class_specifier, namespace_definition
  csharp      → method_declaration, class_declaration, interface_declaration,
                struct_declaration, enum_declaration
```

**Step 2 — Implement `chunk_with_parser`**

```rust
fn chunk_with_parser(&self, file: &SourceFile) -> Vec<Chunk> {
    let Some(cfg) = lang_config(file.language) else {
        return self.chunk_simple(file);
    };

    let mut parser = Parser::new();
    parser.set_language(cfg.ts_language)
          .expect("grammar version mismatch");

    let tree = match parser.parse(&file.content, None) {
        Some(t) => t,
        None => return self.chunk_simple(file), // parse failed → safe fallback
    };

    let mut raw_chunks = Vec::new();
    collect_splittable_nodes(
        tree.root_node(),
        &file.content,
        cfg.splittable_types,
        &mut raw_chunks,
        &file.path.to_string_lossy(),
        file.language,
    );

    if raw_chunks.is_empty() {
        return self.chunk_simple(file);
    }

    let refined = self.refine_large_chunks(raw_chunks, &file.content);
    self.apply_overlap_ts_style(refined)
}
```

**Step 3 — `collect_splittable_nodes` (DFS)**

```rust
fn collect_splittable_nodes(
    node: Node,
    src: &str,
    splittable: &[&str],
    out: &mut Vec<RawChunk>,
    path: &str,
    lang: mccp_core::Language,
) {
    if splittable.contains(&node.kind()) {
        let text = &src[node.byte_range()];
        if !text.trim().is_empty() {
            out.push(RawChunk {
                content: text.to_string(),
                start_line: node.start_position().row + 1,
                end_line: node.end_position().row + 1,
                start_byte: node.start_byte(),
                end_byte: node.end_byte(),
                path: path.to_string(),
                language: lang,
            });
        }
    }
    for child in node.children(&mut node.walk()) {
        collect_splittable_nodes(child, src, splittable, out, path, lang);
    }
}
```

**Step 4 — Refine oversized nodes (line-based sub-chunking)**

Mirror the `splitLargeChunk` logic from `ast-splitter.ts`:

```rust
fn refine_large_chunks(&self, chunks: Vec<RawChunk>, _src: &str) -> Vec<RawChunk> {
    let max = self.config.max_tokens * 4; // ~4 chars/token heuristic
    let mut out = Vec::new();
    for chunk in chunks {
        if chunk.content.len() <= max {
            out.push(chunk);
        } else {
            out.extend(self.split_large_chunk_by_line(chunk, max));
        }
    }
    out
}
```

**Step 5 — Overlap from previous tail**

```rust
fn apply_overlap_ts_style(&self, mut chunks: Vec<RawChunk>) -> Vec<Chunk> {
    let overlap = self.config.overlap_tokens * 4; // chars
    let mut out = Vec::new();
    for (i, raw) in chunks.iter().enumerate() {
        let content = if i > 0 && overlap > 0 {
            let prev = &chunks[i - 1].content;
            let tail = &prev[prev.len().saturating_sub(overlap)..];
            format!("{}\n{}", tail, raw.content)
        } else {
            raw.content.clone()
        };
        out.push(Chunk::new(
            /* project_id */ String::new(), // filled by pipeline
            raw.path.clone(),
            content,
            raw.start_byte, raw.end_byte,
            raw.start_line, raw.end_line,
            ChunkScope::Method,
        ));
    }
    out
}
```

## Area 2 — MerkleDAG Incremental Sync

### Current gap

`crates/mccp-indexer/src/pipeline.rs` uses a `dashmap::DashMap<String, String>` (`file_hash_cache`) — a flat file→SHA-256 map. To detect changes, the entire map must be walked and compared. For a 50 000-file project this is 50 000 hash lookups on every polling cycle.

### Reference algorithm (`merkle.ts` + `synchronizer.ts`)

The MerkleDAG approach adds a single **root node** whose data is the concatenation of all sorted file hashes. If the root hash is identical to the previous snapshot, **zero file comparisons** are needed — the O(1) root check gates the expensive O(n) file diff.

```
MerkleDAG layout:
  root_node  (sha256 of all child hashes concatenated, sorted by path)
    ├── node(path_a:hash_a)
    ├── node(path_b:hash_b)
    └── …

Compare two snapshots:
  1. Compare root.hash  →  equal → no changes, stop
  2. Compare individual file nodes  →  set diff for added/removed/modified
```

### Migration steps

**Step 1 — Add `MerkleSnapshot` type to `mccp-indexer`**

```rust
// crates/mccp-indexer/src/merkle.rs  (new file)

use sha2::{Sha256, Digest};
use std::collections::BTreeMap;  // BTreeMap ensures sorted iteration

/// Lightweight snapshot of a directory state used for O(1) change detection.
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct MerkleSnapshot {
    /// sha256 of all (path:hash) pairs concatenated in sorted path order.
    pub root_hash: String,
    /// file_path → sha256(content) for every tracked file.
    pub file_hashes: BTreeMap<String, String>,
}

impl MerkleSnapshot {
    pub fn build(file_hashes: BTreeMap<String, String>) -> Self {
        let combined: String = file_hashes
            .iter()
            .map(|(p, h)| format!("{}:{}", p, h))
            .collect::<Vec<_>>()
            .join("");
        let root_hash = format!("{:x}", Sha256::digest(combined.as_bytes()));
        Self { root_hash, file_hashes }
    }

    pub fn diff(&self, new: &MerkleSnapshot)
        -> Option<SnapshotDiff>
    {
        if self.root_hash == new.root_hash {
            return None; // identical — O(1) exit
        }
        let mut added = vec![];
        let mut removed = vec![];
        let mut modified = vec![];

        for (path, new_hash) in &new.file_hashes {
            match self.file_hashes.get(path) {
                None => added.push(path.clone()),
                Some(old_hash) if old_hash != new_hash => modified.push(path.clone()),
                _ => {}
            }
        }
        for path in self.file_hashes.keys() {
            if !new.file_hashes.contains_key(path) {
                removed.push(path.clone());
            }
        }
        Some(SnapshotDiff { added, removed, modified })
    }
}

#[derive(Debug)]
pub struct SnapshotDiff {
    pub added: Vec<String>,
    pub removed: Vec<String>,
    pub modified: Vec<String>,
}
```

**Step 2 — Replace `file_hash_cache` in `IndexingPipeline`**

```rust

//   file_hash_cache: dashmap::DashMap<String, String>,
// With:
```

**Step 3 — Persist snapshot alongside project metadata**

Snapshot is serialised to:
```
~/.mccp/data/mccp/{project_id}/snapshot.json
```

On startup, load the snapshot from disk. On each watcher cycle or `mccp index` call:
1. Walk the project directory → build `BTreeMap<String, SHA256>`
2. Call `MerkleSnapshot::build(new_hashes)`
3. Call `old_snapshot.diff(&new_snapshot)` — returns `None` (no change) or `Some(diff)`
4. Process only the files in `diff.added + diff.modified`, delete chunks for `diff.removed`
5. Save the new snapshot to disk

**Step 4 — Ignore patterns before hashing**

The snapshot walker must respect the same ignore patterns as the indexer (`.gitignore`, etc. — see Area 4). Only hash files that would be indexed.

**Files to change:**
- `crates/mccp-indexer/src/` — add `merkle.rs`, modify `pipeline.rs`
- `crates/mccp-storage/src/persistence.rs` — add `load_snapshot` / `save_snapshot` helpers

## Area 3 — Hybrid Search (Dense + Sparse / BM25)

### Current gap

`crates/mccp-core/src/query.rs` and `VectorStoreProvider` trait only model dense ANN search:
```rust
async fn search(&self, project_id: &str, query: &[f32], top_k: usize, filters: &Filters)
    -> Result<Vec<ScoredChunk>>;
```

Symbol names like `UserService` or `validateToken` are **out-of-vocabulary** for an embedding model — semantic search returns poor results for them. Sparse BM25 search nails exact symbol matches.

### Reference algorithm (Milvus hybrid path)

claude-context creates two vector fields per collection:
- `vector` — dense embeddings (OpenAI/Ollama)
- `sparse_vector` — BM25 sparse embeddings (generated by Milvus's built-in BM25 function at insert time)

At query time it fires two ANN requests in parallel (dense + sparse) and fuses them with **RRF (Reciprocal Rank Fusion)**:
```
rrf_score(d) = Σ_i  1 / (k + rank_i(d))     # k = 60 default
```
RRF is rank-order-only — no score normalisation needed.

### Migration steps

## Area 4 — `.gitignore`-Aware File Filtering

### Current gap

`crates/mccp-indexer/src/pipeline.rs` uses a hardcoded extension list and does not read `.gitignore`, `.dockerignore`, or other project-level ignore files. This means:
- `node_modules/` is indexed unless manually excluded in config
- Generated files (`dist/`, `target/`, `*.min.js`) are indexed

### Reference algorithm (`context.ts` `loadIgnorePatterns`)

The reference implementation:
1. Reads any `.*ignore` file found in the project root (`.gitignore`, `.npmignore`, `.dockerignore`, `.eslintignore`, `.mccpignore`)
2. Reads a global `~/.context/.contextignore`
3. Merges all patterns (deduped) into the traversal filter

```typescript
private async findIgnoreFiles(codebasePath: string): Promise<string[]> {
    const entries = await fs.promises.readdir(codebasePath, { withFileTypes: true });
    return entries
        .filter(e => e.isFile() && e.name.startsWith('.') && e.name.endsWith('ignore'))
        .map(e => path.join(codebasePath, e.name));
}
```

### Migration steps

**Step 1 — Add `ignore` crate**

```toml
# Cargo.toml
ignore = "0.4"    # walkdir + gitignore aware traversal (same crate used by ripgrep)
```

**Step 2 — Replace manual file traversal in `pipeline.rs`**

```rust
use ignore::WalkBuilder;

fn walk_project_files(root: &Path, extra_patterns: &[String]) -> impl Iterator<Item = PathBuf> {
    let mut builder = WalkBuilder::new(root);
    builder
        .hidden(true)           // respect .gitignore
        .git_ignore(true)       // automatic .gitignore support
        .git_global(true)       // ~/.config/git/ignore
        .git_exclude(true);     // .git/info/exclude

    // additional project-level patterns from config
    for p in extra_patterns {
        builder.add_ignore(p);
    }

    builder.build()
        .filter_map(|e| e.ok())
        .filter(|e| e.file_type().map(|t| t.is_file()).unwrap_or(false))
        .map(|e| e.into_path())
}
```

The `ignore` crate is battle-tested (it powers `ripgrep` and `fd`). It handles all `.gitignore` semantics including negation patterns and directory-only rules.

**Step 3 — Global ignore file**

Support `~/.mccp/ignore` as a global ignore list:

```rust

    dirs::home_dir().map(|h| h.join(".mccp").join("ignore"))
}
```

## Area 5 — Embedding Dimension Auto-Detection

### Current gap

`config.toml` requires `dimensions = 768` to be set manually. If a user switches embedding models (e.g. from `nomic-embed-text` at 768d to `mxbai-embed-large` at 1024d) and forgets to update the config, vectors are silently wrong.

### Reference algorithm (`ollama-embedding.ts`)

```typescript
async embed(text: string): Promise<EmbeddingVector> {
    if (!this.dimensionDetected && !this.config.dimension) {
        this.dimension = await this.detectDimension();
        this.dimensionDetected = true;
    }
    // ...
}

async detectDimension(testText = "test"): Promise<number> {
    const response = await this.client.embed({ model: this.config.model, input: testText });
    return response.embeddings[0].length;
}
```

Known model defaults are short-circuited: `nomic-embed-text → 8192 tokens, 768 dims`.

### Migration steps

**Step 1 — Add dimension probe to `EmbeddingProvider` trait**

```rust
#[async_trait]

    async fn embed(&self, texts: &[String]) -> Result<Vec<Vec<f32>>>;
    async fn embed_one(&self, text: &str) -> Result<Vec<f32>>;
    fn dimensions(&self) -> usize;
    async fn probe_dimensions(&self) -> Result<usize> {

```

**Step 2 — Auto-detect on provider init**

In `ProviderManager::init()`, after constructing the embedding provider:

```rust

}
```

## Area 6 — Progress Callback / Streaming Index Status

### Current gap

`IndexingPipeline::start()` has no way to report per-file progress. The TUI and CLI `mccp index status` command have no real-time feed.

### Reference algorithm (`context.ts`)

```typescript
async indexCodebase(
    codebasePath: string,
    progressCallback?: (p: { phase: string; current: number; total: number; percentage: number }) => void,
    forceReindex: boolean = false
): Promise<{ indexedFiles: number; totalChunks: number; status: 'completed' | 'limit_reached' }>
```

The callback fires at:
- `Preparing collection…` (0%)
- `Scanning files…` (5%)
- `Processing files (N/M)…` (10–100%, linearly per file)
- `Indexing complete!` (100%)

```rust
        percentage: pct as u8,
    }).ok();
}
```

**Step 4 — Expose in MCP server and TUI**

The MCP server (`crates/mccp-server/src/server.rs`) can subscribe to `progress_rx` and push SSE events. The TUI (`src/interactive.rs`) subscribes and renders the progress bar in the index queue panel.

- `crates/mccp-server/src/server.rs` — subscribe and push SSE
- `src/interactive.rs` — render progress

## Additional Improvements (Lower Priority)

### A — Sparse BM25 corpus statistics persistence

The BM25 encoder requires corpus-level term statistics (DF per term, total document count). These must be:
- Persisted to `~/.mccp/data/mccp/{project_id}/bm25_stats.bin`
- Updated incrementally on file add/modify/remove (not full rebuild)
- Invalidated when the project is reset (`mccp index --reset`)

Recommended approach: use a compact inverted index backed by `sled` (already a dependency) with atomic delta updates per file change event.

### B — Cross-encoder re-ranking (optional)

AGENT.md specifies an optional re-ranking step using an Ollama rerank model. After RRF fusion returns `top_k * 2` candidates, a cross-encoder scores each `(query, chunk)` pair. Add this as an opt-in config:

```toml
[query]
reranker = "mxbai-rerank-base-v1"   # leave empty to disable
```

This should be behind a feature flag — it doubles latency for small gains on most queries.

### C — `mccp index --status` real-time output

Currently `mccp index status` is described in AGENT.md but `crates/mccp-cli/src/commands.rs` doesn't implement it. With Area 6's `progress_rx` in place, the CLI command becomes:

```rust
// subscribe to progress_rx via HTTP GET http://localhost:7422/index/status SSE stream
// render with ratatui progress bar or plain text
```

### D — Query cache warming on daemon start

On daemon restart, the LRU query cache is cold. Cache warming can be implemented by replaying the last N queries from the query log at startup time (background task, low priority). This mirrors the "summary cache lazy loading from disk" spec in AGENT.md.
